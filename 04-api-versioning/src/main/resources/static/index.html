üìö About This Module

Learn how to version your REST APIs to maintain backwards compatibility while evolving your application. This module demonstrates URL-based versioning strategy with two API versions showing real-world evolution patterns.

üéØ Key Learning: How to evolve an API from a simple user model (V1) to an enhanced model (V2) without breaking existing clients.

üîÑ API Evolution: V1 ‚Üí V2

üì¶ V1 API (Original)

User Model:

  • id - User ID
  • name - Full name (single field)
  • email - Email address

Use case: Legacy clients, simple requirements

üöÄ V2 API (Enhanced)

User Model:

  • id - User ID
  • firstName - First name (split)
  • lastName - Last name (split)
  • email - Email address
  • phone - Phone number (NEW)

Use case: New clients, enhanced features

üîå V1 API Endpoints

Base URL: http://localhost:8084/api/v1

GET /api/v1/users

Get all users (V1 format)

curl http://localhost:8084/api/v1/users
GET /api/v1/users/{id}

Get user by ID (V1 format)

curl http://localhost:8084/api/v1/users/1
POST /api/v1/users

Create user (V1 format)

curl -X POST http://localhost:8084/api/v1/users \
  -H "Content-Type: application/json" \
  -d '{
    "name": "John Doe",
    "email": "john@example.com"
  }'

üöÄ V2 API Endpoints (Enhanced)

Base URL: http://localhost:8084/api/v2

GET /api/v2/users

Get all users (V2 enhanced format)

curl http://localhost:8084/api/v2/users
GET /api/v2/users/{id}

Get user by ID (V2 enhanced format)

curl http://localhost:8084/api/v2/users/1
POST /api/v2/users

Create user (V2 enhanced format)

curl -X POST http://localhost:8084/api/v2/users \
  -H "Content-Type: application/json" \
  -d '{
    "firstName": "John",
    "lastName": "Doe",
    "email": "john@example.com",
    "phone": "555-1234"
  }'
GET /api/v2/users/search?firstName={name}

Search by first name (V2 only feature)

curl "http://localhost:8084/api/v2/users/search?firstName=John"

üß™ How to Test

  1. Test V1: Create a user using V1 endpoint (simple name field)
  2. Test V2: Create a user using V2 endpoint (firstName/lastName fields)
  3. Compare: GET users from both endpoints - see different response formats
  4. V2 Feature: Try the search endpoint (only available in V2)
  5. Verify: Both versions work simultaneously - no breaking changes!

üìñ Key Concepts Demonstrated

  • URL-based Versioning: /api/v1/* vs /api/v2/*
  • Backwards Compatibility: V1 continues working while V2 is available
  • API Evolution: Adding fields without breaking existing clients
  • Java Records: UserDTO and UserDTOV2 as immutable data carriers
  • Controller Organization: Separate controllers per version
  • Feature Additions: V2-only endpoints (search)
