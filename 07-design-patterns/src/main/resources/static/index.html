Factory Pattern

Creational - Object creation

Create objects without specifying exact classes. Spring's BeanFactory is a perfect example.

@Component
public class NotificationFactory {
    public Notification create(String type)
}

Use: Flexible object creation

üîå

Strategy Pattern

Behavioral - Algorithm selection

Define family of algorithms, encapsulate each, make them interchangeable at runtime.

interface PaymentStrategy {
    void pay(BigDecimal amount);
}

Spring: Used in dependency injection

üëÄ

Observer Pattern

Behavioral - Event notification

One-to-many dependency. When one object changes state, all dependents are notified.

@EventListener
public void handleOrder(
    OrderEvent event)

Spring: ApplicationEvent framework

üé≠

Proxy Pattern

Structural - Controlled access

Provide surrogate/placeholder for another object. Spring AOP uses this heavily.

@Transactional
@Cacheable
public User findById(Long id)

Spring: AOP, @Transactional, @Cacheable

Template Method

Behavioral - Algorithm skeleton

Define skeleton of algorithm, let subclasses override specific steps.

abstract class DataProcessor {
    final void process() {
        load(); transform(); save();
    }
}

Spring: JdbcTemplate, RestTemplate

Singleton Pattern

Creational - Single instance

Ensure class has only one instance. Spring beans are singletons by default.

@Component // Singleton by default
@Scope("singleton")
public class ConfigService

Spring: Default bean scope

üìñ Key Concepts

  • Gang of Four: 23 classic design patterns from 1994 book
  • SOLID Principles: Foundation for good design patterns
  • Dependency Injection: Spring's core pattern (Strategy + Factory)
  • AOP: Aspect-Oriented Programming uses Proxy pattern
  • Template Method: Spring's *Template classes
  • Event-Driven: Observer pattern in Spring

üéì When to Use Patterns

  1. Factory: When object creation logic is complex
  2. Strategy: When you have multiple algorithms for same task
  3. Observer: When changes in one object affect others
  4. Singleton: When exactly one instance is needed (use Spring beans!)
  5. Template Method: When algorithm structure is fixed but steps vary
  6. Proxy: When you need to control access or add behavior
